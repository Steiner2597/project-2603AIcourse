# 探索与精修两阶段专项解析

## 1. 这份文档讲什么

这是一份专门给后续接手同学看的流程文档，只讲一件事：

FunSearch-Lite 里的两阶段机制到底是什么、在代码哪里发生、在实验日志里怎么看、为什么它能起作用。

并且使用两个真实案例贯穿：

- 单 DS 模型案例：runs/ds-整体实验/260325115953_or_ds
- DS+GPT 协作案例：runs/260325234133_or_ds+gpt

本文目标是“通俗易懂 + 可证据追溯”。

---

## 2. 先用一句大白话解释两阶段

可以把每一代每个岛的 LLM 工作理解成两步：

1. 探索阶段（generate / proposal）
- 作用：多提新思路，扩大搜索面。
- 特征：偏发散，关注提出新候选表达式。

2. 精修阶段（refine）
- 作用：围绕当前表现较好的表达式做定向优化。
- 特征：偏收敛，关注把已有好思路做细化改进。

如果只探索不精修，容易“想法很多但不够稳”。
如果只精修不探索，容易“早熟收敛、卡局部最优”。
两阶段结合，本质是发散与收敛并行。

---

## 3. 两阶段在代码中的真实位置

核心在 src/funsearch_lite/evolution.py 的每代每岛流程中：

1. 先设置 proposal 上下文并调用 generate_candidates。
2. 再取 top-k 精英，设置 refine 上下文并调用 refine_topk。

也就是严格顺序：

- proposal -> refine

对应调用链：

- CLI 参数进入 src/funsearch_lite/cli.py
- 构造 llm_client 与 search_config
- run_experiment 启动引擎
- IslandEvolution._evolve_one_generation 中依次触发两阶段
- llm_client 把每次请求写入 checkpoints/requests.jsonl

所以两阶段不是“概念层面”，而是每一代每个岛都明确调用两次 LLM。

---

## 4. 每代每岛的完整微流程（建议背下来）

在某一代某一岛，实际执行顺序是：

1. 保留一部分精英（上一代好的候选）。
2. 按重启策略注入随机候选（防止过早同质化）。
3. 探索阶段：LLM generate 候选。
4. 精修阶段：LLM refine top-k 精英。
5. 混合 LLM 候选、变异、交叉生成候选池。
6. 对每个候选做评估（真实装箱结果）。
7. 通过表达式去重 + 行为去重 + 行为限流过滤无效重复。
8. 做多样性选择，形成下一代。

你可以理解为：

- 探索和精修负责“出招”。
- 评估与去重负责“判招”。
- 选择与重启负责“留招和补招”。

---

## 5. 如何从日志判断两阶段是否真的在运行

看 checkpoints/requests.jsonl 就够了。

判定规则：

1. 每个代-岛组合，应出现一对请求：
- request_type = generate, phase = proposal
- request_type = refine, phase = refine

2. 顺序应为先 generate 再 refine。

3. 在 summary.json 里，generation >= 1 时，每个岛 llm_requests 通常等于 2。

4. generation = 0 初始化阶段，llm_requests 应为 0。

这 4 条同时满足，基本可确认两阶段机制在正常执行。

## 6. 案例 A：单 DS 模型（完整 8 代）

案例目录：runs/ds-整体实验/260325115953_or_ds

### 6.1 关键信息

- 任务：OR-Library combined
- generations = 8
- islands = 2
- llm_seed_per_generation = 2
- llm_refine_top_k = 2
- provider_tag = ds

### 6.2 两阶段证据

在 requests.jsonl 开头就能看到：

- request_id 1: generate + proposal
- request_id 2: refine + refine

同一代另一岛也是同样顺序。

### 6.3 请求总数核对

理论请求数：

- 2(阶段) * 8(代) * 2(岛) = 32

summary.json 实际：

- llm_request_start = 32
- llm_request_done = 32
- llm_request_error = 0

一致。

### 6.4 结果侧观察

metrics.json 显示：

- 最终 avg_gap_ratio = 0.05552689068865102（约 5.55%）
- best_per_gen 从 0.061289 降到 0.055949（进化结束）
- local search 再把 best_score 从 0.055949 降到 0.055527

结论：

两阶段 + 常规进化 + 局部搜索形成了完整闭环，最终结果优于 BF/FF 基线。

## 7. 案例 B：DS+GPT 协作（3 代）

案例目录：runs/260325234133_or_ds+gpt

### 7.1 关键信息

- 任务：OR-Library combined
- generations = 3
- islands = 2
- llm_seed_per_generation = 1
- llm_refine_top_k = 1
- provider_tag = ds+gpt

### 7.2 两阶段分工证据（非常直观）

requests.jsonl 明确记录了 provider 字段：

- generate 阶段使用 deepseek-chat，provider = ds
- refine 阶段使用 gpt-5-nano，provider = gpt

这说明协作模式不仅是“名字上混合”，而是请求级别真分工。

### 7.3 请求总数核对

理论请求数：

- 2(阶段) * 3(代) * 2(岛) = 12

summary.json 实际：

- llm_request_start = 12
- llm_request_done = 12
- llm_request_error = 0

一致。

### 7.4 结果侧观察

metrics.json 显示：

- 最终 avg_gap_ratio = 0.060997616847007795（约 6.10%）
- local search 从 0.061289 改进到 0.060998

这次混合 run 仍优于 BF/FF，但数值不如案例 A 的单 DS run。

这里要强调：

这两个 run 参数不同（代数、种群、llm_seeds、topk 都不同），不能当作严格公平对比，只能当“流程展示案例”。

## 8. 单 DS 与 DS+GPT 的流程差异，到底差在哪

两者相同点：

1. 两阶段时序完全相同：先探索后精修。
2. 每代每岛都有两次 LLM 调用。
3. 后续评估、去重、选择流程一致。

两者不同点：

1. 模型路由不同。
- 单 DS：两阶段都走 DS。
- DS+GPT：探索走 DS，精修走 GPT。

2. 单次请求耗时结构不同。
- DS+GPT 案例中 refine（GPT）耗时明显更长，平均可到 20-40 秒区间。
- 单 DS 案例中 generate/refine 都在 DS，耗时分布更接近。

3. 参数规模不同会影响结果解释。
- 3 代小预算 run 的上限与 8 代 run 不同。

---

## 9. 为什么这个两阶段机制通常有效

从搜索理论看，它同时解决了两个矛盾：

1. 要探索新区域，但又不能每代都“推倒重来”。
2. 要稳步改进，但又不能一直围着旧表达式打转。

探索阶段给“新方向”，精修阶段给“深挖能力”，再加上进化机制的筛选与去重，就形成了稳定提升路径。

你可以把它看成：

- 探索负责找到潜在山头。
- 精修负责爬高当前山头。
- 多岛+迁移负责避免所有人挤在同一座山。

## 10. 接手同学最常用的排查手册

### 问题 1：我怀疑 refine 没执行

检查：

1. requests.jsonl 是否有 request_type=refine。
2. phase 是否为 refine。
3. 每代每岛 llm_requests 是否接近 2。

### 问题 2：我怀疑协作路由没生效

检查：

1. requests.jsonl 里的 provider 字段。
2. generate 是否确实是 ds。
3. refine 是否确实是 gpt。

### 问题 3：结果不稳定

先看三项：

1. 是否只有单 seed。
2. 是否 generations 太少。
3. 是否 behavior 去重阈值过松导致同质化。

## 11. 复现实验的建议模板

### 11.1 单 DS 两阶段

- 使用你已有的 DS 常规参数模板。
- 固定 seed，先跑 1 个，再扩展到 3 个 seed。

### 11.2 DS+GPT 协作两阶段

- generate provider 设为 ds。
- refine provider 设为 gpt。
- 保持其余参数与单 DS 对齐，才可做公平对比。

建议做公平实验时遵循：

- 同数据集
- 同 generations/population/islands
- 同 llm_seeds/topk
- 同随机种子集合

## 12. 一段可直接放进报告的说明文字

“本系统在每个 generation 的每个 island 中采用两阶段 LLM 协作流程：先执行探索阶段（generate/proposal）提出候选表达式，再执行精修阶段（refine）对 Top-K 精英表达式进行定向优化。两阶段候选与变异、交叉、随机重启共同构成候选池，随后经过评估、去重与多样性选择进入下一代。该流程可在 requests.jsonl 中通过成对的 generate/refine 事件验证，并可在 summary.json 中通过 llm_requests 统计进行一致性核对。”

---

## 13. 最后总结

如果你只记住三件事，就记这三条：

1. 两阶段不是抽象概念，而是每代每岛的固定执行步骤。
2. 是否真的执行，必须用 requests.jsonl 与 summary.json 交叉验证。
3. 单 DS 与 DS+GPT 的核心差异在“阶段路由”，不是流程顺序。
