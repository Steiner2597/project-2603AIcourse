# FunSearch-Lite 代码交接全景解析（面向后续接手同学）

## 1. 文档目标

这份文档的目标是让新同学在不依赖口头说明的前提下，完整理解以下内容：

- 项目是如何从命令行参数一路执行到最终指标与图表的。
- “探索阶段（generate/proposal）”与“精修阶段（refine）”在代码与实验日志中的真实落点。
- 如何读懂 run 目录下每个产物文件。
- 如何定位问题、复现实验、做公平对比，以及安全扩展新功能。

本文使用以下真实实验作为主示例：

- runs/ds-整体实验/260325115953_or_ds

## 2. 一句话架构

这是一个“LLM 候选提案 + 进化搜索筛选 + 基线评估对比 + 可观测日志/检查点”的在线装箱启发式搜索系统。

主调用链：

1. CLI 解析参数与 provider/model 路由。
2. 构造 SearchConfig/ExperimentConfig。
3. 进入 run_experiment 管理 run 目录、状态、数据集与引擎。
4. IslandEvolution 执行多岛进化：探索 -> 精修 -> 变异/交叉/重启 -> 评估去重 -> 选择。
5. 局部搜索后处理最优表达式。
6. 产出 metrics、曲线、比较表、summary、checkpoint。

核心入口文件：

- src/funsearch_lite/cli.py
- src/funsearch_lite/experiment.py
- src/funsearch_lite/evolution.py
- src/funsearch_lite/llm_client.py
- src/funsearch_lite/evaluator.py
- src/funsearch_lite/compare.py
- src/funsearch_lite/summarize.py
- src/funsearch_lite/visualize.py

## 3. 端到端执行流程（按真实代码顺序）

### 3.1 CLI 层：参数解析与模型路由

文件：src/funsearch_lite/cli.py

职责：

- 解析所有搜索参数（generations/population/islands/去重阈值/停滞参数等）。
- 解析模型参数：api、gen-provider、ref-provider、gen-model、ref-model。
- 支持 ds 双模型快捷开关 --ds-dual-model（生成=deepseek-chat，精修=deepseek-reasoner）。
- 构造 SearchConfig 与 ExperimentConfig。
- 根据 --compare 或 --orlib-mode 分支进入 run_experiment 或 generate_comparison。

关键点：

- --ds-dual-model 仅改变“模型路由和模型名”，不会自动覆盖你其他搜索超参数。
- 未显式传入的参数会回落到默认值（需警惕对照实验公平性）。

### 3.2 实验编排层：run_experiment

文件：src/funsearch_lite/experiment.py

职责：

- 创建 run 目录和 checkpoints 子目录。
- 写 status.json（running -> completed/failed）。
- 加载数据集：OR-Library 或 synthetic。
- 调用 IslandEvolution.run() 完成主进化。
- 执行 local search（对 top-k 精英进行微扰优化）。
- 计算最终 evaluation 与 FF/BF baseline。
- 写入 metrics.json / best_expression.txt / curve.png。
- 调用 summarize 自动产出 summary.json、summary.md、baseline_compare.png。

### 3.3 搜索引擎层：IslandEvolution

文件：src/funsearch_lite/evolution.py

职责：

- 初始化多岛种群（随机表达式 + 去重 + 多样性筛选）。
- 每代逐岛执行演化。
- 可选停滞检测触发探索增强（restart 比例和 llm seeds 临时增大）。
- 持续写 generation checkpoint 与 population checkpoint。

每代每岛的逻辑顺序：

1. 保留部分精英。
2. 执行随机重启注入（按 restart 参数）。
3. LLM 探索（proposal/generate）。
4. LLM 精修（refine，对 top-k 精英定向改写）。
5. 用 LLM 候选 + 变异/交叉补齐候选池。
6. 候选评估、表达式去重、行为去重、行为限流。
7. 多样性筛选后进入下一代。

### 3.4 LLM 客户端层：请求与日志

文件：src/funsearch_lite/llm_client.py

职责：

- 按 generate/refine 路由到对应 provider/client/model。
- 发送 chat.completions 请求。
- 记录请求日志到 checkpoints/requests.jsonl。
- 支持 timeout/retry/backoff。
- 对 reasoner 模型做 temperature 兼容处理。

### 3.5 评估层：真实装箱与指标

文件：src/funsearch_lite/evaluator.py

职责：

- 对每个实例执行在线装箱决策（按表达式 scorer 打分选箱）。
- 计算 bins_used、gap、gap_ratio。
- 生成行为签名（全轨迹采样 + bins_used）并哈希。

主指标：

- OR-Library：avg_gap_ratio（越小越好）。
- 无最优解场景：avg_bins_used（越小越好）。

### 3.6 汇总与可视化层

文件：

- src/funsearch_lite/summarize.py
- src/funsearch_lite/visualize.py
- src/funsearch_lite/compare.py

职责：

- 写 summary.json / summary.md（按代、按岛统计 eval 与 LLM 请求）。
- 生成 baseline_compare.png（FF/BF/Ours 柱状图）。
- 生成 curve.png（best/avg 轨迹，x 轴从 1 开始）。
- 生成 comparison_table.md/csv（含 vs BF/FF 的改进列）。

## 4. “探索阶段 vs 精修阶段”在实验中的真实落点

以下内容直接对应示例 run：runs/ds-整体实验/260325115953_or_ds。

### 4.1 在日志中的证据

文件：runs/ds-整体实验/260325115953_or_ds/checkpoints/requests.jsonl

你会看到严格成对出现的事件：

- request_type=generate 且 phase=proposal
- request_type=refine 且 phase=refine

并且顺序是每代每岛先 generate 再 refine。

例如 generation=1, island=0：

- 先 request_id=1（generate/proposal）
- 后 request_id=2（refine/refine）

generation=1, island=1 同样如此（request_id=3,4）。

### 4.2 在汇总统计中的证据

文件：runs/ds-整体实验/260325115953_or_ds/summary.json

可观察到：

- generation 0 的 llm_requests=0（初始化阶段，不走 LLM）。
- generation >=1 后，每个岛 llm_requests=2。

对这个 run：

- generations=8
- islands=2
- 理论 LLM 请求数 = 8 * 2 * 2 = 32
- summary.json 中 llm_request_start=32，完全一致。

这可以视为“探索+精修双阶段被完整执行”的强证据。

### 4.3 在配置上的触发逻辑

文件：runs/ds-整体实验/260325115953_or_ds/metrics.json

search_config 显示：

- llm_seed_per_generation=2
- llm_refine_top_k=2

含义：

- 探索阶段每次最多取 2 个候选。
- 精修阶段从当前岛前 2 名精英表达式出发，再产出最多 2 个候选。

## 6. run 目录产物“怎么读”

以 runs/ds-整体实验/260325115953_or_ds 为例：

- best_expression.txt：最终最优表达式（提交/复现的核心对象）。
- metrics.json：最关键主文件（配置、历史轨迹、最终评估、基线、局部搜索结果）。
- comparison_table.md/csv：报告中的方法对比表。
- summary.json/md：按 generation/island 的过程诊断报表。
- baseline_compare.png：最终方法对比柱状图。
- curve.png：进化轨迹图（best 与 avg）。
- checkpoints/gen_XXXX.json：代级快照（快查收敛趋势）。
- checkpoints/requests.jsonl：LLM 请求粒度日志（排查超时、重试、阶段路由最有用）。
- checkpoints/eval_events.jsonl：候选评估事件日志（排查去重命中、错误、来源配比最有用）。
- checkpoints/population_gen_XXXX.json：每代完整种群表达式快照（用于复盘/调参分析）。
- checkpoints/local_search.json：局部搜索是否改进，以及每一步状态。
- checkpoints/binpackX.json 与 binpacks/binpackX/report.json：分测试集结果。

## 7. 参数如何映射到流程

### 7.1 影响探索/精修“数量”

- llm-seeds：每阶段生成候选数。
- llm-topk：精修阶段使用的精英数量。
- islands/generations：决定双阶段调用次数总量。

粗略预算公式（无停滞额外加成时）：

- LLM 请求数 ≈ 2 * generations * islands

### 7.2 影响“探索强度”

- restart-ratio、restart-interval
- stagnation-*（停滞触发后增强）
- oversample-ratio

### 7.3 影响“同质化抑制”

- behavior-repeat-cap
- behavior-eval-cap
- behavior-signature-points

### 7.4 影响后期“收敛偏好”

- anneal
- novelty-weight-final
- stability-weight-final
- behavior-repeat-cap-final
- behavior-eval-cap-final

### 7.5 影响最终小幅冲刺

- local-topk
- local-steps

## 8. 常见误区与排查建议

### 误区 1：ds-dual-model 会自动带上你之前所有调参

不是。它只改变模型路由与模型名。其他参数若不显式传入会用默认值。

建议：做对照实验时把关键搜索参数全部显式写在命令里。

### 误区 2：生成/精修是否真的都在跑

看 requests.jsonl：

- 是否同时出现 request_type=generate 和 request_type=refine。
- 是否按代按岛成对出现。

### 误区 3：只看单次 seed 就下结论

不建议。至少 3 个 seed，报告均值和标准差。

### 误区 4：曲线波动就是退化

不一定。avg 波动可能来自多样化探索，需结合 best 曲线与最终 local search 判断。

## 9. 建议的交接验证清单（新同学第一天）

1. 跑一次最小 smoke test（1-3 代）。
2. 检查 run 目录是否完整产出 metrics/summary/checkpoints。
3. 打开 requests.jsonl，确认 generate/refine 双阶段均有记录。
4. 对同一参数跑 2-3 个 seed，确认结果可重复且波动合理。
5. 对比 FF/BF/Ours 三者相对位置是否符合预期。

## 10. 扩展开发建议（避免破坏稳定性）

### 10.1 新增模型协作策略

优先改动：

- src/funsearch_lite/cli.py（新增参数与路由逻辑）
- src/funsearch_lite/llm_client.py（请求协议与日志字段）

必须保证：

- requests.jsonl 仍能区分 request_type、phase、provider、model。
- 失败重试和超时策略不被绕过。

### 10.2 调整搜索策略

优先改动：

- src/funsearch_lite/evolution.py

必须保证：

- generation checkpoint 仍连续可读。
- population checkpoint 仍完整落盘。
- 选择逻辑改动后，summary 统计可解释。

### 10.3 调整评估特征

优先改动：

- src/funsearch_lite/evaluator.py
- src/funsearch_lite/heuristic_expr.py

必须保证：

- ALLOWED_NAMES 与 evaluator features 同步。
- 老表达式不会因缺失变量直接崩溃。

## 11. 附：给后续报告可直接复用的“流程描述”

可以直接复用下面这段：

“在每个 generation 的每个 island 中，系统先执行探索阶段（proposal/generate），由 LLM 基于历史反馈提出行为差异化候选；随后执行精修阶段（refine），对当前 Top-K 精英表达式进行定向改写。两阶段候选与进化算子（变异、交叉、随机重启）共同进入评估池，经过表达式去重、行为去重与行为限流后再做多样性筛选，形成下一代。该流程在 requests.jsonl 中以成对的 generate/refine 事件体现，并可在 summary.json 中由每代每岛 llm_requests=2 得到验证。”

---

## 12. 结语

接手同学只需抓住三件事：

1. 入口在 CLI，真实执行在 experiment + evolution。
2. 核心创新闭环是“探索（generate）+ 精修（refine）”。
3. 一切过程都应能在 requests/eval_events/gen/population/summary 中被证据化。

只要做到“参数显式、日志可证、对照公平”，该项目就能稳定迭代并持续产出可解释结果。
