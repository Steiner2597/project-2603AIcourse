本报告为一篇全新报告，分析以下 3 次整体模式（or_ds）实验：

- runs/260325115953_or_ds
- runs/260325120006_or_ds

目标：

1. 评估整体模式下 FunSearch-Lite 的总体效果与稳定性。

## 5. 分组视角（1-4 vs 5-8）

结合 docs/ORLIB_8测试集特性与调参分析.md 的分组特征：

- binpack1-4（容量 150、混合大/小件）：
  - 三次 run 的分组均值：[4.711, 4.883, 4.937]
  - 均值 4.844，std 0.096（稳定）
- binpack5-8（容量 100、tightness≈1）：
  - 三次 run 的分组均值：[6.394, 5.572, 5.430]
  - 均值 5.799，std 0.425（波动明显更大）

结论：

1. 整体模式在“tightness≈1”的 5-8 组上更难稳定优化。
2. 该组对精细匹配失误更敏感，容易出现 run 间差异放大。

## 8. 与分集模式结果对照（简要）

对照已有分集模式报告（or_parts_ds 三次）：

- 分集模式总体均值：4.6987%

- 当一个表达式要同时覆盖 8 个 source，必然牺牲部分子集最优性。
- per-source 专家化表达式在该任务上具备结构性优势。

## 9. 下一轮调参建议（整体模式专用）

### 9.1 抑制末代回弹

1. 将 restart-ratio 从 0.18 下调到 0.12-0.15。
2. 将 stability-weight-final 从 1.0 上调到 1.1-1.3。
3. generations 可增至 10，并采用“末两代弱扰动”策略。

### 9.2 提高 binpack5-8 稳定性

1. 保持 behavior-repeat-cap=1，尝试 behavior-eval-cap=2。
2. 增强 local search（top_k 2→3，steps 2→3）。
3. 对 binpack5 增加小样本复评，降低偶然性影响。

### 9.3 管控系统长尾延迟

1. 对单次 LLM 请求设置软超时与重试上限。
2. 将高延迟请求记录到独立日志，和分代性能联动分析。
3. 在相同配置下重复运行 5 次，剔除系统侧抖动后再比较参数优劣。

---

## 10. 结论

1. 整体模式下，FunSearch-Lite 三次运行全部优于 BF/FF，平均相对提升达到 14.85%（vs BF）与 16.71%（vs FF）。
2. 主要瓶颈在“多 source 折中”与“tightness≈1 子集波动”，尤其 binpack5-8 稳定性不足。
3. 与分集模式对照显示，per-source 专家化策略仍是当前更优路径。
4. 若坚持整体模式，优先优化方向应是：
   - 后期扰动控制（减少回弹）
   - 对 5-8 组增强微调与复评
   - 降低 LLM 长尾时延对演化节奏的影响。
